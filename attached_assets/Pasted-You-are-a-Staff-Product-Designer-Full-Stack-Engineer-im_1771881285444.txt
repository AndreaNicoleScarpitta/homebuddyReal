This is an EXTENSION of the existing “Systems Section” on the Dashboard. Do NOT redesign navigation, onboarding, or the overall IA.

GOAL
Right now users can add systems but cannot assess or manage them. Fix this by providing:
1) a clear summary of how many systems exist by type
2) a frictionless “review existing vs add new” decision when selecting a system type
3) a dedicated, easy place to view all systems and their details

CONSTRAINTS (NON-NEGOTIABLE)

- Preserve current Dashboard as the hub.

FEATURES TO ADD

1) SYSTEMS INVENTORY SUMMARY (Dashboard → Systems Section)
On the Dashboard Systems section, add a compact inventory header:
- “Your Systems” + total count
- Per-system-type counts (e.g., HVAC (2), Plumbing (1), Electrical (1), Roof (1), Appliances (3))
Display as small chips or inline pills using existing design tokens.

Behavior:
- If count is 0 for a type, show it as “Add” not a number (optional).
- Clicking a type should route into the type’s “Instances” view (see below).

2) SYSTEM TYPE “INSTANCES” VIEW (Review vs Add)
When a user selects a system TYPE (e.g., HVAC), do NOT immediately jump to “Add new”.
Instead show:

A) If 0 instances exist:
- Empty state with short explanation
- Primary CTA: [Add HVAC System]

B) If 1+ instances exist:
- Show list of existing instances (cards)
- At top: “You have X HVAC systems”
- Primary CTA: [Add Another HVAC System]
- Secondary: [Review All] (if list is truncated)

Each instance card must show a scan-friendly summary:
- Name (Auto-generated or user-editable)
- Key fields (brand/model/age where available)
- Last updated
- Quick actions: [Review] and optional [Edit]

This accomplishes “prompted to review existing or add new”.

3) SYSTEMS DIRECTORY (Easy way to view ALL systems)
Provide one consolidated “All Systems” view that lists every system instance across types.

Placement requirement:
- Prefer placing this as a link/button at the top of the Dashboard Systems section:
  [View All Systems]
This opens a directory page/view.

Directory Requirements:
- Filter by type (chips)
- Search by keyword (brand/model/nickname)
- Sort: most recently updated (default), type, oldest
- Each row/card shows:
  - Type icon
  - Instance name
  - Key attributes
  - Count of related tasks (if available)
  - Link: [Open]

Do NOT make this a new nav item unless absolutely necessary. If you must add it, add it as a sub-item under the existing Dashboard/Overview, not a new top-level section.

4) SYSTEM DETAIL VIEW (Instance Detail Page)
When the user clicks a specific instance, show a detail page that aggregates “information related to it”:

Required sections:
- Overview card (type, nickname, location in home optional, brand/model/age)
- Generated maintenance tasks (list) with statuses
- Recent maintenance history entries related to this system (if relationship exists; otherwise show a placeholder + “Link to system” option)
- Documents linked to this system (if relationship exists; otherwise allow attaching)
- Notes field (optional, user-owned)

Keep scope tight:
- Do NOT build an entire CMMS.
- Just make systems reviewable and actionable.
